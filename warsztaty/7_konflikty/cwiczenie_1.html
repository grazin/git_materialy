Pracujesz nad projektem razem z zespołem. Strategia branching/merging się sprawdza, teraz przyjdzie się wam zmierzyć z konfliktami. Zaczynasz nową funkcjonalność, opisaną w Jirze w tasku issue-13. Po skończonej pracy przygotuj swój brancz tak, żeby był aktualny względem brancza master i zawierał tylko jeden komit.

Użyj repozytorium z poprzedniego ćwiczenia.

$ git checkout master
$ git pull
$ git checkout -b issue-13-[your-github-username]

Aby rozpocząć pracę, upewnij się, że masz najnowszą wersję brancza master, a następnie odbranczuj się z niego z unikalną nazwą issue-13-[your-github-username].

W pliku data/config.json zmień dane w obiekcie json.
W pliku src/file1.txt zmień linijkę pierwszą.
Zakomituj zmiany i wypuszuj do origina.

W tak zwanym międzyczasie koledzy z zespołu zrobili zmianę na branczu master w tych samych plikach.
Plik data/config.json powinien nadal mieć zawartość taką jak na branczu master,
natomiast plik src/file1.txt powinien zawrzeć zmiany z obu branczy.

$ git checkout master
$ git pull
$ git checkout issue-13-[your-github-username]
$ git rebase master

Po zakończonej pracy rozpocznij ribejz.

$ git diff --name-only --diff-filter=U

Po poleceniu rebase otrzymasz informację o konfliktach. Wylistuj skonfliktowane pliki.

$ git checkout --ours data/config.json

Najpierw rozwiąż konflikt z pliku data/config.json na korzyść brancza master.

Następnie rozwiąż ręcznie konflikty w pliku src/file1.txt tak aby zawierał zmiany z obu branczy.

$ git add .

Teraz oznacz konflikty jako rozwiązane.

$ git rebase --continue

I dokończ ribejz.

$ git log --graph

Sprawdź historię swojego brancza.
