Pracujesz nad projektem razem z zespołem. Postanowiliście udoskonalić strategię branching/merging i stosować rebase interaktywny, żeby każdy feature merdżowany do mastera miał tylko jeden komit. Zaczynasz nową funkcjonalność, opisaną w Jirze w tasku issue-13. Po skończonej pracy przygotuj swój brancz tak, żeby był aktualny względem brancza master i zawierał tylko jeden komit.

$ git clone git@github.com:grazin/rebase_interactive_workshop.git

Sklonuj repozytorium rebase_interactive_workshop z Githuba.

$ git checkout master
$ git pull
$ git checkout -b issue-13-[your-github-username]

Aby rozpocząć pracę, upewnij się, że masz najnowszą wersję brancza master, a następnie odbranczuj się z niego z unikalną nazwą issue-13-[your-github-username].

Dodaj jeden plik i zakomituj zmiany.
Dopisz linijkę na końcu jednego z istniejących już plików i zakomituj zmiany.
Wypuszuj wszystko do origina.

Aktualnie masz na swoim branczu 2 komity. Musisz je złączyć w jeden. Użyj polecenia rebase --interactive, żeby to osiągnąć.

$ git log master..issue-13-[your-github-username]

W pierwszym kroku sprawdź ile masz komitów na swoim branczu (czasami praca nad ficzerem trwa kilka dni i łatwo zapomnieć ile ich się zrobiło).

$ git rebase -i HEAD~2

Po upewnieniu się, że są 2 komity, wejdź w tryb interaktywnego ribejza.

pick 3ac1c09 komit #1
f 50e26c3 komit #2

Wejdź w tryb edycji ( i lub Insert) i przy pierwszym komicie zostaw pick a przy drugim wstaw f (fixup) lub s (squash).

Aby kontuować wciśnij Escape a następnie Shift+zz lub :wq.

$ git log master..issue-13-[your-github-username]

Teraz ponownie sprawdź log. Powinien teraz być jeden komit.

$ git push origin +issue-13-[your-github-username]

Wypuszuj teraz swoje zmiany w takiej formie. Musisz ponownie dodać parametr --force lub + przed nazwą brancza, ponieważ zmieniłeś jego historię.
