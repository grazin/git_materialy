Pracujesz nad projektem razem z zespołem. Postanowiliście zmienić strategię branching/merging i stosować rebase, żeby zachować przejrzystość repozytorium. Zaczynasz nową funkcjonalność, opisaną w Jirze w tasku issue-13. Po skończonej pracy przygotuj swój brancz tak, żeby był aktualny względem brancza master.

$ git clone git@github.com:grazin/rebase_workshop.git

Sklonuj repozytorium rebase_workshop z Githuba.

$ git checkout master
$ git pull
$ git checkout -b issue-13-[your-github-username]

Aby rozpocząć pracę, upewnij się, że masz najnowszą wersję brancza master, a następnie odbranczuj się z niego z unikalną nazwą issue-13-[your-github-username].

Dodaj jeden plik i dopisz linijkę na końcu jednego z istniejących już plików. Zakomituj zmiany i wypuszuj do origina.

W tak zwanym międzyczasie koledzy z zespołu zrobili zmianę na branczu master.

$ git checkout master
$ git pull
$ git checkout issue-13-[your-github-username]
$ git rebase master
$ git push issue-13-[your-github-username] --force
$ git checkout master
$ git merge issue-13-[your-github-username]

Chcesz zmerdżować swoje zmiany do mastera po skończeniu pracy. Najpierw jednak musisz wykonać rebase na swoim branczu.

Parametr --force jest konieczny przy puszowaniu po zmianie historii.

$ git log --graph

Teraz spójrz na graf historii. Lub w git GUI Linia komitów jest prosta.
